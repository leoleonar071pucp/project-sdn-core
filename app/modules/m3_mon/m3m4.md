# Instrucciones para el agente – Creación de la estructura base de M3

## 1. Contexto del módulo M3
M3 es un módulo de monitoreo y detección de ataques de red (volumétricos y escaneos verticales) que se ejecuta en una VM de pruebas (4 vCPU, 4 GB RAM, 15 GB disco). Recibe tráfico espejado a través de una interfaz de red dedicada controlada por DPDK, extrae flujos (clave 6‑tupla) y aplica dos motores de detección con umbrales fijos. Las alertas se envían por HTTP/REST a un M4 externo.

## 2. Configuración previa (NO MODIFICAR)
- **Sistema operativo:** Ubuntu 22.04 LTS.
- **DPDK:** instalado vía `apt` (`dpdk`, `libdpdk-dev`), versión 21.11 LTS.
- **Hugepages:** 128 páginas de 2 MB (256 MB) montadas en `/mnt/huge`.
- **Interfaz de red:** la segunda interfaz (`ens4`) ya está desvinculada del kernel y asignada a `vfio-pci` mediante `dpdk-devbind.py`. El PCI address es `0000:00:08.0` (puede variar; se tomará del archivo de configuración).
- **Paquetes adicionales instalados:** `build-essential`, `pkg-config`, `libnuma-dev`, `libcurl4-openssl-dev`, `libmicrohttpd-dev` (opcional para M4, no necesario para M3), `cjson` (si se usa como librería externa; si no, se incluirá como fuente).
- **NO INSTALAR NADA NUEVO.** Todo el software necesario ya está presente.

## 3. Requerimientos funcionales y técnicos de M3
- **Lenguaje:** C (estándar C11).
- **DPDK:** para ingesta de paquetes (`rte_eal`, `rte_eth_rx_burst`, `rte_hash`, `rte_ring`, etc.).
- **HTTP/REST:** las alertas se envían mediante `libcurl` a un endpoint configurable.
- **JSON:** se usará `cJSON` para construir las alertas en formato JSON (puede ser como fuente incluida en `common/cJSON.c/h` o como dependencia del sistema; se prefiere incluirlo como fuente para evitar dependencias externas).
- **Estructuras de datos:**
  - Tabla de flujos: `rte_hash` con clave de 13 bytes (vlan, ip_src, ip_dst, port_src, port_dst, protocol), datos con contadores atómicos (`packet_count`, `byte_count`, `last_seen`).
  - Tabla de hosts: `rte_hash` con clave de 6 bytes (vlan, ip_src), datos con filtro Bloom global y hasta 5 destinos con filtros Bloom individuales (LRU).
  - Filtro de Bloom: 2048 bits (256 bytes), 4 funciones hash (FNV-1a con semilla).
- **Motores de detección:**
  - Volumétrico (cada 1 s): contar flujos por destino (> 100 en pruebas), flujos efímeros sin respuesta (> 10), ratio unidireccional > 0.8 (ventana 10 s).
  - Escaneo vertical (cada 5 min): cardinalidad estimada > 15 puertos a un destino, > 50 puertos global.
- **Umbrales configurables** en `config.yaml`.
- **Hilos:** ingesta (DPDK RX), evaluador (recorre tablas y aplica motores), publicación de alertas (consumidor de anillo `rte_ring`).

## 5. Descripción de cada archivo y su contenido mínimo

### 5.1. `common/alert_schema.h`
- Define la estructura `alert_t` con campos: `event_id` (uuid), `timestamp`, `anomaly_type` (enum), `attacker_ip`, `attacker_vlan`, `target_ip`, `target_port`, `protocol`, `measured_pps`, `measured_bps`, `confidence`, `severity`.
- También define una función `alert_to_json(alert_t *a, char *buf, size_t len)` (o se implementa en un `.c` aparte).

### 5.2. `common/rest_helper.h`
- Proporciona `int send_alert(const char *json_str, const char *endpoint)` que usa libcurl para hacer POST.
- Incluye inicialización de libcurl (que puede hacerse en `main` o de forma perezosa).

### 5.3. `m3/config.yaml`
```yaml
dpdk:
  pci_address: "0000:00:08.0"
  mbuf_pool_size: 8192
thresholds:
  flood_dst_flows: 100
  fast_scan_src_flows: 10
  unidir_ratio: 0.8
  vertical_scan_dst_ports: 15
  global_scan_ports: 50
idle_timeout_s: 5
host_idle_timeout_s: 300
max_flows: 100000
max_hosts: 5000
alert_endpoint: "http://localhost:8080/alerts"
```

### 5.4. m3/src/config.h
Constantes y variables globales leídas de config.yaml (o se pueden cargar en main y pasar a los módulos).

Declaraciones de extern para los umbrales y tamaños.

### 5.5. m3/src/main.c
Inicializa DPDK (rte_eal_init).

Lee config.yaml (usando una biblioteca YAML simple como libyaml o parseo manual básico; en pruebas se puede hardcodear o usar config.h).

Configura el puerto DPDK con rte_eth_dev_configure y rte_eth_rx_queue_setup.

Crea la tabla de flujos y tabla de hosts (llamando a flow_table_init, host_table_init).

Lanza el hilo de captura (capture_loop).

Lanza el hilo evaluador (detection_loop).

Lanza el hilo de publicación de alertas (que consume de un rte_ring y llama a send_alert).

Espera señales o bucle infinito.

### 5.6. m3/src/capture.c y capture.h
capture_loop: función ejecutada en un hilo dedicado.

Extrae paquetes con rte_eth_rx_burst.

Para cada paquete, parsea las cabeceras (VLAN, IP, TCP/UDP/ICMP) usando funciones DPDK.

Construye las claves y llama a flow_table_update y host_table_update.

Libera los mbufs.

Exporta estadísticas opcionales (contador de paquetes).

### 5.7. m3/src/flow_table.c y flow_table.h
flow_table_init(max_flows): crea la tabla rte_hash con los parámetros adecuados.

flow_table_update(flow_key *key, uint16_t len): realiza lookup y, si existe, incrementa atómicamente packet_count y byte_count y actualiza last_seen; si no, intenta insertar con contadores a 0. Si la tabla está llena, incrementa un contador global de descartes.

flow_table_get_stats: devuelve estadísticas (opcional).

Estructura flow_key y flow_entry.

### 5.8. m3/src/host_table.c y host_table.h
host_table_init(max_hosts): crea la tabla rte_hash para hosts.

host_table_update(host_key *key, uint32_t dst_ip, uint16_t dst_port, uint64_t now): obtiene o crea entrada, actualiza last_seen, inserta en filtro Bloom global y en el filtro del destino (búsqueda LRU en array de 5).

host_table_evaluate(uint64_t now): recorre entradas, elimina inactivas (>300 s), estima cardinalidades y genera alertas (invoca a detection.c o directamente emite alertas).

Estructuras host_key, host_entry, per_dst_entry.

### 5.9. m3/src/bloom.c y bloom.h
bloom_init(bloom_filter_t *bf): pone a cero los 256 bytes.

bloom_insert(bloom_filter_t *bf, uint16_t port): aplica 4 hashes FNV-1a con semillas y pone los bits correspondientes a 1 con __atomic_fetch_or.

bloom_estimate(const bloom_filter_t *bf): cuenta bits 1, aplica fórmula de máxima verosimilitud y devuelve double.

#define BLOOM_SIZE_BITS 2048, etc.

### 5.10. m3/src/detection.c y detection.h
detection_volumetric(flow_record *records, int count): recibe una lista de registros de flujo del último segundo, aplica los umbrales volumétricos y genera alertas.

detection_scan(host_entry *host): llamada desde host_table_evaluate, aplica umbrales de escaneo y genera alertas.

Funciones auxiliares para ventana deslizante unidireccional (puede guardar estado en el módulo).

Las alertas se encolan en un rte_ring para el hilo de publicación.

### 5.11. m3/src/alert.c y alert.h
alert_init(ring): inicializa el anillo.

alert_enqueue(alert_t *): inserta una alerta en el anillo.

alert_publisher_loop: hilo que extrae alertas del anillo, las convierte a JSON con cJSON, y llama a send_alert del rest_helper.h.

alert_t definido en alert_schema.h.

### 5.12. m3/meson.build

#### 6. Lo que el agente NO debe hacer
NO instalar software adicional.

NO modificar configuraciones del sistema (hugepages, drivers, etc.).

NO alterar la interfaz de red asignada a DPDK.

NO usar ZeroMQ o cualquier otro bus de mensajería; solo REST con libcurl.

NO implementar M4 ni el servidor REST de M4; solo el cliente HTTP en M3.

NO crear los archivos de M4 (solo la estructura de common/ y m3/).


#### 7. Secuencia de acciones para el agente
Crear las carpetas ~/m3m4/common, ~/m3m4/m3/src.

Verificar los archivos cJSON.c y cJSON.h en common/.

Crear common/alert_schema.h y common/rest_helper.h con las definiciones mínimas.

Crear m3/config.yaml con los valores de prueba.

Crear todos los archivos .c y .h en m3/ según las descripciones anteriores, con implementaciones mínimas:

En main.c, inicializar DPDK y crear las tablas, lanzar hilos con pthread (o usar las primitivas de DPDK). Los hilos pueden ser funciones dummy al principio.

En capture.c, un bucle que captura paquetes y los cuenta, pero sin procesar aún (para comprobar que funciona).

En flow_table.c, implementar init y update como funciones vacías que devuelvan éxito.

Así sucesivamente.

Crear m3/meson.build.

Compilar: cd m3 && meson build && cd build && ninja. Debería compilar sin errores.

Ejecutar: sudo ./m3 -l 0-1 -- -m 256 y verificar que se inicia y captura paquetes (imprimiendo contadores).

#### 8. Criterios de aceptación
El proyecto compila limpiamente.

config.yaml se puede modificar para cambiar umbrales.

El binario m3 captura paquetes de la interfaz DPDK y muestra PPS.

Se puede añadir una alerta dummy que se envíe al endpoint configurado y se reciba en un servidor de prueba (p.ej., nc -l 8080).
