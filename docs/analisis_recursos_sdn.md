# Dimensionamiento de Recursos Computacionales - SDN Core

Este documento presenta el dimensionamiento técnico de la arquitectura SDN, estructurado específicamente para cumplir con los criterios de evaluación. Cada componente detalla sus recursos asignados, su plataforma y una **justificación técnica exhaustiva**. Esta versión detallada sirve como respaldo profundo para defender la arquitectura ante paneles técnicos y jefes de práctica (JPs).

---

## 1. Entorno de Laboratorio / Demo (Prueba Piloto)
Este entorno emulado (Mininet/OVS) busca probar la viabilidad técnica y los escenarios de prueba funcionales.
- **Carga de trabajo esperada:** 5 a 15 usuarios concurrentes emulados, con picos de hasta 50 dispositivos realizando solicitudes simultáneas de red e inicios de sesión.

### Tabla de Dimensionamiento - Nodos de Gestión, Control y Datos

| Identificador del Componente | Función en la Solución | SO / Plataforma | vCPU | RAM | Disco | Justificación Exhaustiva de la Asignación (Carga de 50 usuarios) |
|---|---|---|:---:|:---:|:---:|---|
| **VM 1: Auth & Autho**<br>(M1, M2, RADIUS, MySQL) | Gestión de identidades, evaluación de políticas RBAC (OPA) y validación criptográfica de credenciales. | **Ubuntu 18.04 LTS**<br>+ Docker | **2** | **4 GB** | **8 GB** | <ul><li>**SO / Plataforma:** Se eligió Ubuntu 18.04 LTS (CLI) por ser el estándar estable instalado en los laboratorios; al no tener interfaz gráfica, reserva sus recursos puramente para servicios en segundo plano. Docker asegura aislamiento para OPA.</li><li>**vCPU (2):** El inicio de sesión es bloqueante. Si usamos 1 núcleo, cuando MySQL hace una lectura intensiva, FreeRADIUS y el Portal se "congelan" esperando. Con 2 núcleos, un núcleo atiende exclusivamente el I/O de base de datos mientras el otro procesa la criptografía de contraseñas y las peticiones web simultáneas de los 50 usuarios, evitando errores de *Timeout*.</li><li>**RAM (4 GB):** Desglose crítico: El SO base consume ~512 MB. MySQL recibe ~1 GB para su *InnoDB Buffer Pool* para cargar la tabla de usuarios entera en memoria y evitar ralentizaciones por leer el disco. Docker y OPA requieren ~1 GB para cargar el árbol JSON (`data_completo.json`) en memoria. El saldo (~1.5 GB) es un colchón de seguridad obligatorio para que los picos de carga no activen el *OOM Killer* (mecanismo que apagaría el servidor si la RAM llega al 100%).</li><li>**Disco (8 GB):** La instalación base del SO ocupa ~3 GB. Las imágenes de contenedores Docker (M2 y OPA) ocupan ~3 GB. Dejamos 2 GB libres para el crecimiento local de MySQL (transacciones temporales). Más disco sería desperdicio, y menos disco causaría corrupción si se satura.</li></ul> |
| **VM 2: Monitoreo y Detección**<br>(M3, M4, M5) | Inspección profunda de paquetes (DPI), detección de amenazas en tiempo real y recolección de logs (Auditoría). | **Ubuntu 18.04 LTS** | **2** | **4 GB** | **8 GB** | <ul><li>**SO / Plataforma:** Ubuntu LTS sin interfaz gráfica, maximizando el rendimiento para la ingesta de texto masivo (logs) de las herramientas de auditoría.</li><li>**vCPU (2):** La captura continua de paquetes (Sniffing) no puede detenerse. Si usáramos 1 solo núcleo y la máquina se ocupara procesando una alerta o indexando un log, la tarjeta de red "dejaría pasar" paquetes maliciosos sin inspeccionarlos (Blind spots). Los 2 núcleos garantizan lectura ininterrumpida.</li><li>**RAM (4 GB):** Los módulos de Monitoreo (M5) actúan almacenando los flujos temporales de red en memoria antes de escribirlos al disco duro (buffer). Intentar hacer esto con 1 GB o 2 GB causaría fragmentación severa o colapsaría los *scripts* de inspección de Python.</li><li>**Disco (8 GB):** Reducido de los 20GB iniciales por recomendación académica para una demo. El SO ocupa 3 GB. Quedan 5 GB libres dedicados 100% a almacenar temporalmente archivos PCAP (capturas de red) e historial de logs. Es la medida perfecta para una prueba de un par de horas sin llegar a colgar la máquina virtual.</li></ul> |
| **Controlador SDN**<br>(ONOS) | Cerebro de la red. Descubre topología, calcula enrutamiento e instala flujos OpenFlow enviados por M6. | **Ubuntu 18.04 LTS**<br>JVM (Java) | **2** | **4 GB** | **8 GB** | <ul><li>**SO / Plataforma:** Requiere la Máquina Virtual de Java (JVM) corriendo sobre Ubuntu, ya que ONOS está programado en este lenguaje orientado a grandes flujos de red.</li><li>**vCPU (2):** El paralelismo es vital en ONOS. El núcleo 1 mantiene "vivas" las sesiones TCP/OpenFlow (saludos y *keep-alives*) de los 5 switches en la red. El núcleo 2 se dedica matemáticamente a compilar las reglas M6 (Intent Framework) usando el algoritmo de Dijkstra. Con 1 núcleo, los pings de la red experimentarían latencias masivas.</li><li>**RAM (4 GB):** Este es el cuello de botella más clásico de los controladores SDN. Java exige asignar una reserva rígida (Heap Size) al arrancar; a ONOS se le asignan 2 GB puros (`-Xms2G`). Si el controlador tuviese menos RAM general, el recolector de basura de Java consumiría toda la CPU tratando de liberar megabytes, trabando toda la red del campus simulado.</li><li>**Disco (8 GB):** ONOS es un software in-memory. Su base de datos distribuida guarda todo el estado de la red directamente en la RAM. Los 8 GB solo sirven para alojar el sistema operativo y los binarios comprimidos de instalación (.tar.gz), sobrando espacio.</li></ul> |
| **Switches de Red**<br>(SW1 a SW5) | Plano de datos emulado. Conmutación de paquetes según reglas inyectadas por el controlador. | **Ubuntu 18.04 LTS**<br>Open vSwitch | **1** | **1 GB** | **3 GB** | <ul><li>**SO / Plataforma:** Open vSwitch (OVS) opera como un módulo inyectado directamente en el kernel de Linux (Ubuntu), haciéndolo extremadamente veloz.</li><li>**vCPU (1):** En las redes SDN, el switch es "tonto". Toda la inteligencia recae en el Controlador (ONOS). Por ende, el switch solo necesita un núcleo para consultar su tabla de flujos interna. No requiere paralelismo.</li><li>**RAM (1 GB):** 1 GB es la recomendación oficial de Ubuntu Server para prevenir que procesos del sistema recurran al "Swap" (paginación en disco). Usar Swap destruiría el ancho de banda y la velocidad de conmutación.</li><li>**Disco (3 GB):** Es el peso base del sistema operativo. Los switches no almacenan información histórica ni corren contenedores.</li></ul> |
| **Nodos Finales**<br>(H1 a H4, Servidores) | Emulación de dispositivos de usuario (alumnos) y servidores de recursos destino. | **Ubuntu 18.04 LTS**<br>(CLI) | **1** | **1 GB** | **3 GB** | <ul><li>**SO / Plataforma:** Emulados nativamente vía Mininet u OS ligeros para generar tráfico TCP/UDP válido.</li><li>**vCPU/RAM/Disco:** Son hosts "zombies" cuyo único fin es enviar comandos (`ping`, `curl`) al Portal Cautivo para probar las restricciones M2. Con 1 vCPU y 1 GB de RAM se asegura que el sistema responda ágilmente a la terminal sin consumir recursos innecesarios del hipervisor del laboratorio.</li></ul> |

## 2. Criterios de Escalabilidad: Entorno de Producción Universitario
Para evidenciar que la arquitectura puede evolucionar sin requerir cambios sustanciales en su diseño lógico, se proyecta el dimensionamiento hacia una red real de campus para **10,000 a 25,000 conexiones**. 

*(Nota: La lógica del código M1/M2/M6 se mantiene idéntica, solo escala de manera vertical/horizontal el hardware subyacente de los clústeres).*

### Tabla de Proyección - Clúster de Producción

| Identificador del Componente | Función en la Solución | SO / Plataforma | vCPU | RAM | Disco | Justificación Exhaustiva de Escalabilidad (Carga de 25k usuarios) |
|---|---|---|:---:|:---:|:---:|---|
| **Clúster Base de Datos**<br>(MySQL Galera) | Persistencia de sesiones, usuarios y políticas RBAC con alta disponibilidad. | **Linux Enterprise**<br>(RHEL / Ubuntu) | **16** | **64 GB** | **1 TB NVMe** | <ul><li>**Escalabilidad vCPU (16):** La carga máxima (hora de entrada: 8:00 AM) genera bloqueos transaccionales por miles de logins concurrentes. Se requiere un núcleo por cada "pool" de conexiones paralelo.</li><li>**Escalabilidad RAM (64 GB):** Obligatorio para asegurar latencias menores a 1ms. El *InnoDB Buffer Pool* de MySQL se escala para que la tabla completa de los 25,000 alumnos viva 100% en la RAM, eliminando el lentísimo acceso al disco para las consultas de validación.</li><li>**Escalabilidad Disco (1 TB):** Para grabar y retener el "Historial de Sesiones" exigido por las leyes de telecomunicaciones universitarias por hasta 1 año. Obligatorio uso de unidades de estado sólido de servidor (NVMe) que soporten más de 50,000 IOPS (Operaciones I/O por segundo).</li></ul> |
| **Clúster Lógica y Autho**<br>(M1, M2 OPA, RADIUS) | Criptografía EAP, motor de políticas OPA y API de Portal Cautivo. | **Linux Enterprise**<br>Kubernetes / Docker | **8** | **16 GB** | **50 GB** | <ul><li>**Escalabilidad vCPU (8):** Escalado masivo puramente para computación matemática (Hashing criptográfico MD5/SHA en RADIUS) y deserialización web JSON masiva de la API (OPA).</li><li>**Escalabilidad RAM (16 GB):** Se destina a levantar múltiples *workers* de Python paralelos en contenedores (balanceo de carga). Las bases de políticas de M2 se replican en la memoria de cada contenedor de OPA para respuesta inmediata.</li></ul> |
| **Monitoreo/Observabilidad**<br>(M3, M4, M5 Logs) | Ingesta masiva de syslogs (Elasticsearch) e inspección DPI. | **Linux Enterprise**<br>Elastic Stack | **24+** | **128 GB+** | **2 TB+ SSD** | <ul><li>**Escalabilidad Global:** Implementar analítica en tiempo real (Big Data) para encontrar anomalías en una red de 25k dispositivos exige bases de datos vectoriales gigantes (Elasticsearch). Estas consumen toda la RAM disponible para mantener los "índices invertidos" en memoria, y discos de varios terabytes porque una universidad genera millones de líneas de Syslog de red (switches y RADIUS) diariamente.</li></ul> |
| **Clúster Controlador SDN**<br>(3 a 5 nodos ONOS) | Sincronización de estado global (Atomix) y cálculo de algoritmos SPF. | **Linux Enterprise**<br>JVM Cluster | **16**<br>*(c/u)* | **64 GB**<br>*(c/u)* | **250 GB**<br>*(c/u)* | <ul><li>**Escalabilidad Global:** ONOS pasa de ser 1 sola VM a un clúster de alta disponibilidad de al menos 3 servidores físicos. Guardar el estado de miles de equipos interconectados requiere ~48 GB asignados estrictamente a la JVM. Un fallo de RAM (*Out of Memory*) en el controlador central apagaría la conectividad del campus entero.</li></ul> |
| **Plano de Datos**<br>(Core y Distribución) | Conmutación de hardware y enrutamiento físico (Bare-metal). | **ONL / ArubaOS-CX**<br>Switches OpenFlow | **0** | **0** | **0** | <ul><li>**Justificación Cero Recursos Virtuales:** En el salto a producción masiva, ya no existen los "switches de software" (OVS). La conmutación ocurre a 40 Gbps - 100 Gbps utilizando **hardware físico dedicado** con chips integrados ASIC y memoria TCAM de alta velocidad (Ej. Hardware Edgecore, Pica8 o Cisco). Es decir, el plano de datos deja de consumir CPU o RAM de nuestros servidores informáticos.</li></ul> |
